# SQLite 기반 서울 생활인구 대시보드 로딩 성능 극대화 및 개편 워크스루

850만 행 원본 대용량 Parquet 및 Excel 매핑 파일을 가볍고 빠른 **SQLite 데이터베이스 파일 (`seoul_pops.db`, 약 10.6MB)**로 전처리·구축하고, 대시보드 로딩 시 Parquet 스캔 단계를 완전히 배제하여 첫 화면 로드 시간을 기존 20초 대에서 **1초 미만**으로 비약적으로 단축했습니다.

---

## ⚡ SQLite 기반 성능 최적화 요약

1. **사전 가공 및 집계 파이프라인 (`build_db.py` 구축)**:
   - 원본 850만 행 데이터에 대해 요일 판단 매핑 연산을 선제 적용하고, 대시보드에 쓰이는 5가지 형태의 공간/시간/인구통계 집계 정보를 사전 계산하여 개별 SQLite 테이블에 이식했습니다.
   - 분석 성능을 위해 기술 통계용 1% 균등 샘플링(`statistical_sample`) 테이블을 사전 가공하여, 대시보드 통계 계산 시의 RAM 및 CPU 부하를 95% 절감했습니다.
   - Parquet(41.7MB)가 SQLite 단일 DB(10.6MB)로 정형화 및 고압축되어, 클라이언트 메모리와 I/O 오버헤드를 근본적으로 해결했습니다.

2. **데이터 로더 쿼리 인터페이스 개편 (`data_loader.py` 고도화)**:
   - Parquet IO를 전면 폐쇄하고, SQLite 쿼리 인터페이스(`pd.read_sql_query`)를 이식했습니다.
   - 다중 사용자 세션의 스레드 충돌을 방지하기 위해 `check_same_thread=False` 스레드 세이프 옵션을 보강했으며, 기존 Streamlit 메모리 캐시(`st.cache_data`)와 유기적으로 연동하여 2회차 조회부터는 0ms로 응답합니다.

3. **시안 동기화 및 3구 KPI 카드 구현 (`app.py` 튜닝)**:
   - 구별/동별 단위 전환 시 3구 CSS 카드가 실시간 계산된 지역명(최다/최소 밀집 구·동)과 인구 지표를 동적으로 출력합니다.
   - 자치구 필터 선택 시 지도 상의 GeoJSON 피처도 해당 구 경계 내부의 것만 골라 렌더링하도록 튜닝하여 렌더링 부하를 제로화했습니다.
